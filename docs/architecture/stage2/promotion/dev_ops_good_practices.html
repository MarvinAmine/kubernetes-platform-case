Workflow DevOps gouverné

Bonnes pratiques DevOps utilisées

Représentation claire d’un processus de promotion contrôlé, de la pull request jusqu’à la production, avec quality gates, validations QA/PO, déploiement progressif et stratégie de rollback.

Voir le workflow

PR contrôlée

Les PR non conformes sont bloquées avant d’atteindre les environnements supérieurs.

Quality gates

Build, tests, couverture, SonarQube, Snyk et validation humaine sécurisent la livraison.

Go QA / PO

Le QA et le PO donnent leur accord avant les promotions critiques.

Rollback prêt

Si le comportement est instable, le rollback est possible à tout moment.

Chaîne de promotion

Du développement local jusqu’à la production.

Étape 1

Local

Le développeur prépare une feature isolée avec un commit propre et une PR traçable.

Étape 2

Dev

Les checks automatiques bloquent les PR non conformes avant l’intégration.

Étape 3

Staging

Le QA vérifie la feature, les critères d’acceptation et les cas critiques.

Étape 4

Certification

La feature est testée dans un environnement proche de la production.

Étape 5

Production

Déploiement contrôlé, monitoring, validation QA et rollback possible.

Promotion 1

Local -> Dev

PR checks bloquants

Checks automatiques

  • Le build passe.
  • La couverture de tests atteint le seuil attendu, par exemple 85 %.
  • Les tests unitaires et les tests d’intégration passent.
  • Un autre développeur a validé la PR.
  • La PR contient une seule feature à la fois.
  • Le nom de la PR contient le hashtag du ticket Jira.
  • Le nom de la PR commence par [FEATURE], [BUGFIX] ou [PIPELINE].
  • Les checks SonarQube passent.
  • Aucune vulnérabilité critique n’est détectée par Snyk dans les packages.

Checks manuels

  • Vérifier si la feature dépend d’une autre feature qui doit être déployée avant.
  • Vérifier que le feature flag est implémenté lorsque le risque de livraison le justifie.

Objectif

Empêcher qu’un changement non conforme, non testé ou difficile à tracer entre dans l’environnement Dev.

Promotion 2

Dev -> Staging

Une feature à la fois

Critères d’acceptation

Les conditions d’acceptation du récit ont été vérifiées préalablement.

Démo QA

Une démo est faite avec le QA pour valider le happy path et les corner cases critiques vérifiables.

Bug détecté

Si le QA trouve un bug dans la feature, la feature retourne en Dev.

Promotion 3

Staging -> Certification

Validation QA / PO

Validation fonctionnelle

  • Les tests E2E passent.
  • Le QA et le PO donnent leur go pour le déploiement en certif.
  • Le QA teste la feature sur l’environnement de certification.

Communication

Un canal Teams est ouvert pour notifier le PO et le QA de la progression du déploiement.

Pourquoi c’est important

Le canal centralise les décisions, les validations, les incidents et les actions de rollback.

Déploiement progressif / canary-ish

1. Groupe bêta

Un endpoint spécifique est utilisé par des bêta-testeurs ou par une autre équipe interne.

2. Redirection graduelle

Si la nouvelle version fonctionne correctement, le trafic est orienté graduellement vers elle.

3. Rollback

Si le comportement n’est pas satisfaisant, retour immédiat vers la version stable.

Promotion 4

Certification -> Production

Déploiement contrôlé

Contrôle d’accès

  • Seul le DevOps a accès au pipeline et aux credentials de production.
  • Le PO et le QA donnent leur go avant la promotion en production.
  • Le DevOps est réservé à une date spécifique pour le déploiement.

Présence pendant le déploiement

  • Le DevOps et au moins un développeur sont en appel.
  • Idéalement, le développeur présent est celui qui a implémenté la feature.
  • Le QA vérifie le comportement en production.

Déploiement canary en production

1. Version stable

La version actuelle reste disponible comme point de retour.

2. Trafic progressif

Le trafic est graduellement redirigé vers la nouvelle version.

3. Validation QA

Le QA valide les comportements critiques directement en production.

4. Rollback

Si le comportement n’est pas satisfaisant, rollback immédiat.

Communication de production

Un canal Teams est ouvert pour notifier le PO, le QA, le PM et le CIO, au besoin, de la progression du déploiement.

Résumé des décisions

Vision simple des conditions de passage entre environnements.

Transition Contrôle principal Responsables Décision
Local -> Dev Build, tests, SonarQube, Snyk, review, convention de PR Dev + reviewer PR acceptée ou bloquée
Dev -> Staging Critères d’acceptation, démo QA, happy path, cas critiques Dev + QA Promotion ou retour en Dev
Staging -> Certification Tests E2E, validation QA/PO, test en certif QA + PO Go ou no-go
Certification -> Production Accès contrôlé, date réservée, appel de déploiement, validation prod DevOps + Dev + QA + PO Déploiement, canary ou rollback

Vision DevOps résumée

L’objectif n’est pas seulement d’automatiser les déploiements, mais de sécuriser chaque promotion avec des quality gates, des validations humaines ciblées, une traçabilité claire, une communication active et une capacité de rollback rapide.
