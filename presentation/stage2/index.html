OpenShift decision now clear

ADR-016 narrows the scope: Stage 2 uses OpenShift as an enterprise governance target, not as the runtime proof.

Stage 2 on Kubernetes / AKS

Implement the governance outcomes: namespaces, RBAC, quotas, policy guardrails, image promotion, secrets contracts, and observability as code.

OpenShift compatibility checks

Use Sandbox as a compatibility runtime, without pretending it is AKS dev, AKS staging, or a full observability estate.

Compatibility boundary

Make the workload and Kubernetes manifests OpenShift-ready. Keep AKS Terraform, Azure networking, and Azure PostgreSQL as Azure-specific infrastructure.

Stage 3 runtime proof

Move real OpenShift runtime evidence to Stage 3, where hybrid architecture, certification-grade controls, and platform runtime choices belong.

Element OpenShift attention Why it matters
Container image Non-root runtime OpenShift security defaults can reject images that assume root or unsafe filesystem permissions.
Helm chart Portable values The same chart should support AKS and OpenShift without hardcoding cloud-specific assumptions.
Image pulls GHCR pull secret Sandbox or OpenShift projects may need explicit pull-secret wiring for private or rate-limited images.
Exposure Ingress vs Route AKS commonly uses Ingress; OpenShift often uses Route, so exposure should be configurable.
Monitoring ServiceMonitor contract Labels, namespaces, and Prometheus Operator selectors can differ between AKS and OpenShift setups.
Observability stack Available only if Sandbox allows it Prometheus, Grafana, Alertmanager, log collector, Elasticsearch, Kibana, and dashboard ConfigMaps stay in AKS non-prod by default. Draw them in Sandbox only after `oc api-resources`, `oc auth can-i`, and quota checks prove they are available.
Infrastructure Sandbox replacement boundary AKS Terraform, Azure networking, Azure PostgreSQL provisioning, and Azure OIDC stay Azure-specific. Sandbox validates only the workload contract: Project, Route/Service, pull secret, ConfigMap/Secret, and DB connectivity.
Database External first, pod fallback AKS dev and staging use Azure PostgreSQL for prod parity. Sandbox should try external Azure PostgreSQL when networking allows it; PostgreSQL-in-pod is the fallback compatibility proof.

Constraints already locked

These are not optional design ideas anymore. They are the constraints the implementation must consume.

Environment model

Local -> dev -> staging -> prod

Certification is reserved for Stage 3. Stage 2 focuses on controlled promotion and shared-platform governance.

Cluster model

Two-cluster boundary

One non-prod cluster for dev and staging, one isolated prod cluster for production.

Workflow rule

Stage 1 stays versioned

Stage 1 remains replayable through the release tag. `main` evolves to Stage 2 instead of running two active workflow generations.

Promotion rule

Same image across environments

Build once, publish once, promote the same immutable tag or digest through dev, staging, and prod.

OpenShift rule

Aligned outcomes now, runtime later

Stage 2 implements OpenShift-aligned governance on Kubernetes and AKS. Real OpenShift runtime proof moves to Stage 3.

Access rule

Private platform access only

Stage 2 owns private DNS, internal ingress/gateway, and private AKS exposure. Enterprise ZTNA/SASE/VPN integration stays in Stage 3.

What changes by domain

Stage 2 is not one giant refactor. Each area changes for a specific reason tied to the shared-platform model.

Multi-environment foundation

Introduce environment-aware inputs, Terragrunt wiring, state separation, naming, tags, and cluster boundaries.

  • `dev`, `staging`, `prod` inputs
  • Terragrunt for repeated environment wiring
  • two-cluster Azure foundation

OpenShift-aligned guardrails

Implement OpenShift-aligned governance outcomes on Kubernetes / AKS: namespace ownership, policy guardrails, GitOps direction, and secrets governance.

  • project-style namespaces, labels, quotas, and RBAC
  • PSS + Kyverno for SCC-like constraints
  • OpenShift Sandbox checks only where behavior differs
OpenShift-ready Kubernetes Argo CD Vault Kyverno
Application

Promotion-safe packaging

Keep one image, add environment-aware values, and make deployment behavior promotion-safe across dev, staging, and prod.

  • stable runtime contracts
